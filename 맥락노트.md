## 1. 왜 이 아키텍처인가?
==============================

### 1.1 핵심 문제 정의 {#11-핵심-문제-정의}
**전통적 AI 시스템**: 단일 모델, 단일 접근, 제한된 적응성
**우리의 접근**: 다중 에이전트 경쟁, 자가 개선, 적응형 라우팅

### 1.2 설계 철학 {#12-설계-철학}
> "AI를 만드는 것이 아니라, AI를 선택하고 제어하는 것이 우리의 사명"

**근거**:
- 더 나은 AI 모델 구축은 자원 집약적
- 최적 AI로 선택하고 라우팅하는 것이 더 실용적
- 경쟁은 자연 선택을 통해 품질 향상
- 제어는 안전성과 신뢰성 보장

### 1.3 개발에서의 핵심 통찰 {#13-개발에서의-핵심-통찰}

#### 통찰 1: 다중 에이전트 경쟁이 효과적
**관찰**: 다른 정책(refine/balance/expand)이 다른 품질 결과 생성
**결정**: 경쟁 기반 선택 구현
**증거**: 승자 선택이 일관되게 더 나은 출력 생성

#### 통찰 2: 자가 수정이 필수적
**관찰**: 정적 시스템은 변화하는 요구사항에 적응할 수 없음
**결정**: 안전 검사와 함께 런타임 자가 수정 구현
**증거**: 정책 가중치 조정이 성공률 향상

#### 통찰 3: 메모리가 학습 가능하게 함
**관찰**: 메모리 없이 시스템은 실수 반복
**결정**: 지속적 학습 저장 구현
**증거**: 성공 패턴이 시간 경과에 따라 축적

## 2. 아키텍처 결정
==============================

### 2.1 브레인 레이어 vs 바디 레이어 분리 {#21-브레인-레이어-vs-바디-레이어-분리}

**결정**: 분석(브레인)과 실행(바디) 레이어로 분리
**근거**:
- **브레인**: 정적 분석, 계획, 의사결정
- **바디**: 동적 실행, 적응, 학습
- **이점**: 명확한 관심사 분리, 더 쉬운 테스트

**고려된 대안**: 단일 모놀리식 시스템
**거부 이유**: 디버깅 어려움, 모듈성 부족, 진화 어려움

### 2.2 정책 기반 에이전트 시스템 {#22-정책-기반-에이전트-시스템}

**결정**: 세 가지 핵심 정책(refine, balance, expand)
**근거**:
- **refine**: 조직과 명확성
- **balance**: 균형 잡히고 포괄적
- **expand**: 상세와 깊이
- **이점**: 대부분의 응답 유형 효과적으로 커버

**고려된 대안**: 동적 정책 생성
**거부 이유**: 너무 예측 불가능, 제어 어려움

### 2.3 AST 기반 그래프 분석 {#23-ast-기반-그래프-분석}

**결정**: 코드 분석을 위해 Python AST 사용
**근거**:
- **네이티브**: Python 내장, 외부 종속성 없음
- **정확**: 정확한 코드 구조 이해
- **확장 가능**: 의미 분석을 위해 향상 가능

**고려된 대안**: 문자열 기반 파싱
**거부 이유**: 부정확, 취약, 제한된 이해

### 2.4 다중 제공자 AI 통합 {#24-다중-제공자-ai-통합}

**결정**: OpenAI, 로컬 AI, 모의 제공자 지원
**근거**:
- **신뢰성**: 폴백은 시스템 항상 작동 보장
- **비용**: 사용 가능할 때 무료/더 저렴한 옵션 사용
- **개인정보**: 민감 데이터를 위한 로컬 AI

**고려된 대안**: 단일 제공자
**거부 이유**: 단일 실패 지점, 비용 최적화 없음

## 3. 기술적 결정
==============================

### 3.1 Flask vs FastAPI {#31-flask-vs-fastapi}

**결정**: 메인 서버로 Flask 사용
**근거**:
- **단순성**: 이해하고 수정하기 쉬움
- **유연성**: 덜 의견화된 구조
- **호환성**: 기존 코드베이스와 더 나은 호환성

**고려된 대안**: FastAPI
**거부 이유**: 더 복잡, 실험적 작업에 덜 유연

### 3.2 파일 기반 메모리 {#32-파일-기반-메모리}

**결정**: 메모리 저장을 위해 JSON 파일 사용
**근거**:
- **단순성**: 검사 및 디버깅 용이
- **지속성**: 재시작 후에도 생존
- **이식성**: 쉽게 백업 가능

**고려된 대안**: 데이터베이스(SQLite, Redis)
**거부 이유**: 현재 필요에는 과도함, 복잡성 추가

### 3.3 캐싱 전략 {#33-캐싱-전략}

**결정**: 시간 기반 TTL 캐싱
**근거**:
- **단순성**: 구현하기 쉬움
- **효과성**: API 호출 감소
- **안전성**: 데이터 자동 만료

**고려된 대안**: LRU 캐싱
**거부 이유**: 더 복잡, 현재 규모에는 불필요

### 3.4 오류 처리 {#34-오류-처리}

**결정**: 폴백과 함께 우아한 저하
**근거**:
- **신뢰성**: 시스템 계속 작동
- **사용자 경험**: 강한 실패 없음
- **디버깅**: 명확한 오류 로깅

**고려된 대안**: 엄격한 오류 처리
**거부 이유**: 실험적 시스템에 너무 깨지기 쉬움

## 4. 설계 트레이드오프
==============================

### 4.1 성능 vs 유연성 {#41-성능-vs-유연성}

**트레이드오프**: 더 유연한 시스템 vs 더 빠른 성능
**결정**: 유연성 우선
**근거**: 실험적 시스템은 원시 속도보다 적응성이 더 필요

### 4.2 복잡성 vs 유지보수성 {#42-복잡성-vs-유지보수성}

**트레이드오프**: 풍부한 기능 세트 vs 단순한 코드
**결정**: 명확한 분리로 균형
**근거**: 모듈식 설계은 유지보수성 희생 없이 복잡성 허용

### 4.3 안전성 vs 자율성 {#43-안전성-vs-자율성}

**트레이드오프**: 안전한 작업 vs 자가 수정 자유
**결정**: 통제된 자율성과 함께 안전성
**근거**: 자가 수정은 시스템 손상 방지를 위해 안전장치 필요

### 4.4 정확성 vs 속도 {#44-정확성-vs-속도}

**트레이드오프**: 철저한 분석 vs 빠른 응답
**결정**: 시간 제한으로 균형
**근거**: 사용자는 합리적인 응답 시간 기대

### 5.1 초기 그래프 복잡성 {#51-초기-그래프-복잡성}

**문제**: 첫 그래프 버전이 너무 복잡
**교훈**: 단순하게 시작, 복잡성 점진적 추가
**해결책**: 버전화된 접근(v1, v2, v3 등)

### 5.2 과도 엔지니어링 {#52-과도-엔지니어링}

**문제**: 초기 시도가 너무 많은 추상화
**교훈**: YAGNI(너는 그것을 필요로 하지 않을 것이다) 원칙
**해결책**: 실제로 필요한 것만 구현

### 5.3 하드코딩된 종속성 {#53-하드코딩된-종속성}

**문제**: 초기 버전이 하드코딩된 경로
**교훈**: 설정은 외부적이어야 함
**해결책**: 환경 변수와 설정 파일

### 5.4 누락된 오류 처리 {#54-누락된-오류-처리}

**문제**: 초기 버전이 예기치 않은 입력에서 충돌
**교훈**: 예기치 않은 것을 예상
**해결책**: 폴백과 함께 포괄적 오류 처리
